## One-Liner

Design the JSON schema for competitions, rounds, submissions, votes, and leaderboards. Seed with synthetic Rocktober 2024 sample data.

## Problem Statement

The app needs a data model that supports the full competition lifecycle — from setup through daily rounds to final leaderboard — all stored as JSON files in the repo. Time-based scheduling (competition dates, submission windows, voting windows) must be configurable per competition.

## Proposed Solution

1. Define JSON schemas for each entity type under `competitions/<slug>/`
2. `config.json` — competition metadata, members, theme mode, and **schedule configuration** (start/end dates, submission open/close times, voting open/close times, results reveal time)
3. `rounds/day-NN.json` — daily round data: theme, picker, submissions, votes, winner, phase
4. `leaderboard.json` — computed member standings
5. `themeMode` field supports `assigned` (pre-set themes) and `round-robin` (rotating picker from member order)
6. `schemaVersion: 1` in config for future migration path
7. Create synthetic Rocktober 2024 sample data (23 days, 4-5 members) as a reference dataset
8. JSON schema validation script at `tests/validate-schemas.js`

## Acceptance Criteria

**Scenario**: Competition config schema supports full lifecycle configuration
**Given** a new competition is being set up
**When** a `config.json` file is created under `competitions/<slug>/`
**Then** it contains competition metadata, members array, themeMode, and schemaVersion
**And** it includes configurable schedule fields: `startDate`, `endDate`, `submissionOpen`, `submissionClose`, `votingOpen`, `votingClose`, `resultsReveal` (times as HH:MM with timezone)
**And** the `themeMode` field supports both `assigned` and `round-robin` modes

**Scenario**: Round data captures daily competition state
**Given** a competition is active
**When** a round file `rounds/day-NN.json` is created
**Then** it contains theme, themePicker, submissions array, votes array, winner, and phase
**And** the phase field reflects the current round state (submission, voting, results)

**Scenario**: Leaderboard tracks member standings
**Given** rounds have been completed with winners
**When** `leaderboard.json` is computed
**Then** it contains sorted member standings with wins, submissions count, and winRate

**Scenario**: Self-voting is prevented
**Given** a member has submitted a song for the current round
**When** voting opens
**Then** the data model documents that no voter can vote for their own submission

**Scenario**: Sample data validates against schemas
**Given** synthetic Rocktober 2024 sample data exists (23 days, 4-5 members)
**When** the validation script `tests/validate-schemas.js` is run
**Then** all config, round, and leaderboard files pass schema validation
**And** the sample data is stored in the repo for reference and testing
