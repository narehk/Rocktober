## Critical Issues (1)

### [CRITICAL] worker/index.js:53-56 — CORS allows any origin to reflect
```
Context: getCorsHeaders() reflects the request Origin header when CORS_ORIGIN !== '*'
Issue: The condition `allowedOrigin === '*' ? '*' : origin` reflects whatever Origin the
       request sends, rather than checking it against an allowlist. An attacker can set any
       Origin header and it will be reflected in Access-Control-Allow-Origin, defeating the
       purpose of origin restriction. Combined with `CORS_ORIGIN = "*"` in wrangler.toml,
       the /submit endpoint is callable from any domain.
Fix: Either:
  (a) Keep CORS_ORIGIN = "*" (acceptable for a public fun app), OR
  (b) Actually validate: `if (allowedOrigin !== '*' && origin !== allowedOrigin) return 403`
Recommendation: For a fun group project this is acceptable risk. Flag for expert-security
  if the app ever handles real auth tokens or PII.
```

## Errors (2)

### [ERROR] .github/scripts/tally-winner.js:96 — Division by zero when roundsCompleted is 0
```
Context: winRate calculation: `Math.round((s.wins / roundsCompleted) * 100) / 100`
Issue: If no rounds have completed yet (all rounds in submission/voting), roundsCompleted = 0,
       causing division by zero → NaN in leaderboard JSON.
Fix: Guard the division:
  winRate: roundsCompleted > 0
    ? Math.round((s.wins / roundsCompleted) * 100) / 100
    : 0
```

### [ERROR] .github/scripts/tally-winner.js:106 — Hardcoded timezone offset in leaderboard
```
Context: `lastUpdated: latestDate ? `${latestDate}T16:30:00-04:00` : new Date().toISOString()`
Issue: Hardcodes EDT offset (-04:00). During EST (Nov-Mar) this is wrong (-05:00). Also
       assumes all competitions run in the same timezone.
Fix: Use the competition's schedule.timezone from config.json to compute the offset, or
     use ISO UTC format: `${latestDate}T20:30:00Z`
```

## Warnings (5)

### [WARNING] worker/index.js:282 — btoa() may corrupt non-ASCII characters
```
Context: `const encoded = btoa(JSON.stringify(content, null, 2));`
Issue: btoa() only handles Latin-1 characters. If a track title or artist name contains
       emoji or non-Latin characters (common in music), btoa() will throw.
Fix: Use `btoa(unescape(encodeURIComponent(JSON.stringify(content, null, 2))))`
     or use `TextEncoder` + base64 utility.
```

### [WARNING] worker/index.js — No rate limiting on /search or /submit
```
Context: Both endpoints are open to the internet with no throttling.
Issue: A malicious or runaway client could exhaust Spotify API rate limits or spam
       submissions. Low risk for a friend-group app, but worth noting.
Fix: Add Cloudflare rate limiting rules, or a simple in-memory counter per IP.
```

### [WARNING] worker/index.js:207-226 — Race condition on concurrent submissions
```
Context: handleSubmit reads the round JSON, modifies it, then writes it back.
Issue: Two simultaneous submissions could read the same SHA, both write, and the second
       write would fail with a 409 from GitHub Contents API (SHA mismatch). The error is
       not caught specifically — it would surface as a generic "GitHub write failed" error.
Fix: Catch 409 specifically, re-read the file, re-apply the submission, and retry once.
     Or accept the error and let the user retry (acceptable for low-traffic app).
```

### [WARNING] js/app.js:489-494 — innerHTML with template literal interpolation
```
Context: renderSearchResults builds HTML via template literals with escapeHTML()
Issue: The pattern is safe because escapeHTML is used on user-visible strings. However,
       the `data-track` attribute receives JSON that went through escapeHTML, then gets
       JSON.parsed on click — this is fragile. If escapeHTML ever changes behavior or
       a track title contains HTML entities, the parse could fail.
Fix: Consider using data attributes with individual fields instead of serialized JSON,
     or use a DocumentFragment/createElement approach.
```

### [WARNING] .github/scripts/theme-reveal.js:105 — new Date() in Actions uses UTC
```
Context: `const today = new Date();` — used to determine the competition day
Issue: GitHub Actions runners are UTC. The script uses formatDate(today) which now
       correctly uses UTC methods. But if a competition runs in a timezone where the
       date changes (e.g., a competition in Tokyo where UTC midnight is already the
       next day), the round number could be off by one.
Fix: For the current use case (US Eastern competitions) this is fine. If supporting
     international competitions, parse the schedule.timezone from config and compute
     the local date.
```
