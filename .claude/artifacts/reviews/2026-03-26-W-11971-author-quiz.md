## Questions

### Q1: Why are votes stored as both individual records in a votes[] array AND as an incremented sub.votes count, rather than just one or the other?
- A) The votes[] array is for the frontend display, sub.votes is for the backend tally script
- B) votes[] provides an audit trail (who voted for whom), sub.votes maintains backward compat with the existing tally-winner.js that reads sub.votes ✓ (correct)
- C) The dual storage prevents race conditions when two users vote simultaneously ← selected

### Q2: Why does the frontend use an optimistic UI pattern (write localStorage immediately, POST to Worker in background) instead of waiting for the Worker response before updating the UI?
- A) The Worker is unreliable and often fails, so we can't depend on it
- B) Instant feedback — the user sees 'VOTED' immediately without waiting for a network round-trip to the Worker and GitHub API ← selected ✓ (correct)
- C) localStorage is the primary vote store; the Worker POST is optional telemetry

### Q3: The Worker validates votes server-side (phase, membership, duplicates, self-vote). Why not rely solely on the frontend checks that already exist?
- A) Frontend checks are client-side only — they can be bypassed. Server-side validation ensures data integrity regardless of how the request arrives ← selected ✓ (correct)
- B) The Worker runs faster than the frontend, so validation is more efficient there
- C) The frontend doesn't have access to the config.json file, so it can't validate membership

### Q4: What was deliberately left OUT of scope for this implementation?
- A) Vote count display during results phase ← selected
- B) Vote change/retraction — once you vote, you can't switch your vote to a different song ✓ (correct)
- C) Self-vote configuration — the selfVoteAllowed flag was not wired up

## Summary

Strong understanding of security rationale (server-side validation) and UX patterns (optimistic UI). The dual-storage design rationale was missed — the answer focused on race conditions rather than the audit trail + backward compatibility motivation. The scope boundary question was also missed — vote count display was already implemented; the actual exclusion was vote retraction, which is a deliberate simplicity choice for v1.
