## Questions

### Q1: Why was Intl.DateTimeFormat used to resolve UTC offsets in toCompetitionDate() instead of a library like moment-timezone or hardcoded offset tables?
- A) Intl.DateTimeFormat is faster than any third-party library at runtime
- **B) The project has a zero-dependency constraint, and Intl handles DST transitions per-date natively** ✓ ← selected
- C) moment-timezone doesn't support the America/Indiana/Indianapolis timezone

### Q2: Why does handleVote() store votes in localStorage rather than POSTing to a server endpoint?
- A) localStorage is more reliable than network requests for competition integrity
- **B) The Cloudflare Worker (W-11970) doesn't exist yet — localStorage is a future-ready placeholder that W-11970 will extend with a POST** ✓ ← selected
- C) GitHub Pages can't accept POST requests, so client-side storage is the permanent solution
- D) Vote data is too small to justify a server round-trip

### Q3: Why was event delegation (single listener on #submissions-grid) chosen over attaching click handlers to individual vote buttons?
- **A) Event delegation avoids re-attaching listeners when submissions re-render, since the grid container is stable** ✓ (correct)
- B) Individual button listeners would cause memory leaks in older browsers
- C) The submissions grid uses shadow DOM, so direct button access isn't possible ← selected

### Q4: What trade-off does the "user identity via dropdown + localStorage" approach introduce compared to real authentication?
- A) It's slower because localStorage reads block the main thread
- **B) Any user can impersonate another by changing the dropdown — no server-side identity verification** ✓ ← selected
- C) It breaks offline mode because localStorage requires a network connection
- D) It prevents the app from working in incognito/private browsing mode

### Q5: Why does getCurrentPhase() fall back to round.phase for historical rounds instead of always computing from schedule times?
- **A) Historical rounds may have had manual phase overrides or schedule changes that wouldn't be reflected by recomputing from today's config** ✓ (correct)
- B) The toCompetitionDate() function can't handle dates in the past due to DST lookup limitations
- C) Computing phase for past dates would require loading all historical config.json versions from git ← selected

## Summary

Both are nuances that distinguish "it works" from "I know why it works this way."
